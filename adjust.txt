============================================================
NEXT UP — LABELING + EVAL SKILL
============================================================

The remaining 4/7 misses are exactly what an empirical eval loop
should fix. Build a labeling skill (NOT LLM fine-tuning) that:

  1. Captures (brief, mood, style, density, slot, ranked-candidates,
     my-pick) into JSONL fixtures.
  2. Runs all fixtures with current weights → reports agreement %
     per slot, lists disagreements with explanations.
  3. Grid-searches over DEFAULT_RERANK_WEIGHTS to find the weight
     combination that maximizes agreement.

Design questions to discuss BEFORE planning (Phase 0.5 in /develop):

  Q1. Where do fixtures live?
       - agents/scripts/fixtures/*.jsonl
       - .claude/work/fixtures/*.jsonl
       - dedicated agents/eval/ workspace
       Tradeoff: closer to scripts = easier to run; in .claude/work
       keeps it out of production code paths.

  Q2. Interactive vs file-driven labeling?
       - Interactive CLI: script prints candidates, reads stdin pick
       - File-driven: script writes fixture template, user fills in
       Tradeoff: interactive feels nicer but harder to script /
       resume. File-driven plays well with /develop hooks.

  Q3. What does "agreement" mean when my pick isn't in top-3?
       - Strict: only count exact match → harsh
       - Top-3 hit: pick is among top-3 ranked → permissive
       - Rank-weighted: 1.0 for #1, 0.5 for #2, 0.25 for #3 → middle
       Recommend rank-weighted with rank ceiling at K=5.

  Q4. Grid-search granularity?
       - Coarse: 0.0/0.25/0.5/0.75/1.0 per weight, 5 dims = 3125 combos
       - Fine: 0.0..1.0 step 0.1 = 11^5 = ~160k combos
       Coarse is fast (~seconds in memory, no API calls — rerank is
       deterministic given saved candidates). Recommend coarse first.

  Q5. How many fixtures before tuning is meaningful?
       - 5 briefs × 7 slots = 35 labeled picks (minimum)
       - 10 briefs × 7 slots = 70 (better generalization)
       Suggest target: 8-10 briefs spanning bakery, lawyer, SaaS,
       restaurant, agency, e-commerce, fitness, services.

============================================================
BIGGER PICTURE — STAGE ROADMAP (FROM EARLIER CONVERSATION)
============================================================

Stage 0 — DONE: products enum fix, CLI flags for style override.
Stage 1 — End-to-end smoke test, no AWS. Single Node script that
          chains research → style → composer → content → humanizer
          → assembler on a hardcoded brief and writes Next.js
          output to /tmp/site. Mock DDB writes.
Stage 2 — Golden briefs + iterate prompts. The labeling skill above
          is the harness for this stage.
Stage 3 — Make the assembled site browsable. npm run preview.
Stage 4 — AWS plumbing: SQS → Step Functions → Lambdas → DDB →
          Vercel deploy.
Stage 5 — Dashboard + form intake. Seller form → API → SQS →
          pipeline → WebSocket status → preview iframe → deploy.
Stage 6 — Learning loop, QA improvements, cost/latency tuning.

Key principle: don't build orchestration before the units work.
Step Functions, the dashboard, the form intake all FEEL like
progress but they can't tell you whether the pipeline produces
a good site. The smoke-test harness in Stage 1 is what unlocks
the next month of meaningful tuning.
