SiteGen — RAG / Vector DB / Chunking Rollout Plan
===================================================

Task: Multi-stage rollout of RAG, vector DB, embeddings, semantic chunking, and ref-site ingestion across the SiteGen pipeline.
Created: 2026-05-04
Status: ready-for-planning


TASK BOUNDARY
-------------

In scope: Sequencing and shape of a multi-stage rollout that introduces vector retrieval, embeddings, semantic chunking, ref-site scraping, and an LLM-as-judge eval loop into the SiteGen pipeline. Each stage is a self-contained shippable chapter with its own resume narrative.

Out of scope (this discussion):
- Per-stage implementation details (those are decided in /plan per stage)
- Choosing specific Playwright vs Puppeteer, specific anti-bot strategy, specific eval rubric design — captured at planning time for the relevant stage
- Rolling out RAG to Style / Research / SEO agents (deferred — could happen post-Stage-4 if value warrants)
- Multi-modal CLIP embeddings (deferred — not needed for the current chapter set)
- Ingestion Lambda (DDB Stream → re-embed) — deferred until component metadata churn justifies it


IMPLEMENTATION DECISIONS (LOCKED)
---------------------------------

Decision 1 — Foundation-first staging philosophy
Stage 1 is pure infrastructure and tooling. No agent or UI changes. The user explicitly chose foundation first and narrative first over visible product velocity. Each stage maps to one clean resume chapter.

Decision 2 — Stage cadence
Medium stages (~1-2 weeks each), ~4-5 stages total. One coherent feature slice per stage. Each stage is deployable on its own and produces a demoable artifact.

Decision 3 — Vector DB technology: Qdrant Cloud
Managed SaaS. Free tier covers current scale. Resume keyword "Qdrant" lands cleanly without absorbing weeks of self-hosted ops work. Credentials and endpoint live in SSM, accessed via a thin client in agents/shared/.

Decision 4 — Embedding API: OpenAI text-embedding-3-small
1536 dimensions, ~$0.02/M tokens. Industry default. Adds OpenAI as a second LLM vendor (the project currently only uses Anthropic) — API key stored in SSM following the existing /sitegen/dev/claude-api-key pattern.

Decision 5 — First RAG target: Composer Agent
Highest-impact integration point. The Composer's buildUserPrompt already accepts a candidates array — RAG context layers in as a 3rd retrieval source alongside Neo4j graph candidates and DynamoDB fallback candidates. The agent becomes a hybrid graph + vector retriever.

Decision 6 — What gets embedded for Composer RAG
Both:
- Component metadata — joined text of style[] + mood[] + purpose[] + slot text per component
- LLM-generated semantic descriptions — one-shot Claude call per component producing a 1-2 sentence description ("This split hero suits luxury brands needing strong photography"). Generated and embedded in Stage 1's seed script. Resume signal: "used LLMs to enrich component metadata for retrieval."

Decision 7 — Stage 2 success metric: ship-and-watch
No formal eval harness in Stage 2. Watch live generations, tune prompts iteratively. Formal LLM-as-judge eval is the centerpiece of Stage 4, not Stage 2 — keeping it for the final chapter makes Stage 4 a meaningful narrative beat.

Decision 8 — Stage 3 bundles visual + structural scraping
Form field + Playwright Lambda + visual extraction (palette/fonts/screenshot) + HTML semantic chunking + chunk embeddings + RAG retrieval all ship together. ~3 weeks. One resume chapter: "reference-site ingestion with semantic HTML chunking." User accepted the scope risk knowingly.

Fallback if Stage 3 slips: split into Stage 3a (visual only → Style Agent) and Stage 3b (chunking + structural → Composer). Decision made at planning time, not pre-committed.

Decision 9 — Chunking strategy: semantic section chunking (HTML-aware)
Walk the scraped DOM, chunk by <section>, <header>, <main>, semantic block boundaries. One chunk = one logical site section. Most defensible interview answer ("I chunk reference pages by semantic structure to match against component categories"). Lives in agents/shared/chunking.ts. Stage 3 only — no token-window fallback yet (add only if real scraped pages prove unparseable).

Decision 10 — End-state stage: learning loop + LLM-as-judge eval
Stage 4 closes the resume narrative as the "AI engineer" chapter. Tracks seller approve/edit signals, builds an LLM-as-judge rubric pipeline, uses the eval signal to tune retrieval ranking and prompt iterations. ~2-3 weeks.

Decision 11 — Stage 0 added: component metadata audit + enrichment
Inserted before Stage 1. Audit existing 34 component metadata files for gaps and inconsistencies in mood[] / style[] / purpose[]. Enrich where signal is thin. ~3-5 days. Protects every downstream stage — embeddings can only be as good as the underlying tags.

CLAUDE'S DISCRETION
-------------------

- Stage 1 scope composition — User said "decide based on the phases needs." Locked Stage 1 to: VectorStack + embeddings utility + seed script (with LLM description generation folded in). Ingestion Lambda deferred. Chunking helper deferred to Stage 3 (yagni until scraping lands).
- CDK stack placement — VectorStack slots into MainStage between GraphStack and PipelineStack (provides SSM-stored creds that pipeline agents consume). ScraperStack (Stage 3) likely a sibling of PipelineStack or sub-resource depending on whether it's invoked synchronously or via SQS.
- SFN payload field additions — PipelineStack.ts payload blocks for styleStep and composerStep will need updating in Stages 2 and 3 to pass new fields (refSiteUrls, scrapedContext, etc.). Standard chore.
- Embedding cache layer — Stage 1's agents/shared/embeddings.ts will cache by content hash (DDB or S3). Avoid recomputing embeddings on every invocation. Implementation specifics decided at Stage 1 planning.
- Deferred decisions — Local model on Lambda for embeddings (Bedrock Titan, Voyage AI, all-MiniLM) explicitly considered and rejected for Stage 1; can revisit post-Stage-4 if cost or latency becomes a problem.


EXISTING CODE INSIGHTS
----------------------

From scout findings (.claude/work/scout.md):

Greenfield zones (zero existing infra to integrate with):
- No AWS Bedrock usage, no embedding calls, no vector DB references anywhere in the codebase. The only "intelligence layer" beyond LLMs is the Neo4j graph (already live).
- No web scraping libraries, no chunking utilities. Stage 3 is fully greenfield.

Clean RAG injection points (no refactor needed):
- agents/composer/prompt.ts::buildUserPrompt already takes a candidates array as a discrete parameter — adding a ragContext?: string parameter is non-breaking. Composer already does retrieval (Neo4j → DDB fallback), so RAG fits as a third source rather than a replacement.
- agents/content/prompt.ts::buildSystemPrompt is a plain string-returning function. Adding ragExamples?: string is a clean extension when Content gets RAG (post-Stage-4).
- Content Agent currently doesn't even consume researchOutput despite it being in PipelineStateSchema — there's a free-win opportunity for whichever stage touches Content.

Pipeline contract surface:
- ProjectBriefSchema (agents/pipeline-starter/types.ts) — authoritative SQS contract. Add refSiteUrls?: string[] here in Stage 3.
- PipelineStateSchema (agents/shared/types.ts) — full accumulated state. Add refSiteUrls, scrapedContext (visual signals), and ragRetrievals (per-agent retrieved chunks) at relevant stages.
- PipelineStack.ts styleStep and composerStep payload blocks enumerate fields by name — adding any new optional field requires updating both the pipeline-starter null-seed block AND those payload objects.

Component data:
- 34 component metadata files in components/library/<category>/<ComponentName>/metadata.json
- Aggregated manifest at components/library/manifest.json
- Seeded into DDB via agents/scripts/seed-components.ts — Stage 0 enrichment regenerates this; Stage 1 seed script consumes it as input.
- ComponentItem DDB shape has no embedding or vector field. Stage 1 stores embeddings in Qdrant only — DDB stays metadata-only. Embeddings keyed in Qdrant by componentId.

CDK pattern:
- MainStage.ts orchestrates DatabaseStack → GraphStack → PipelineStack → ApiStack → DashboardStack. New stacks slot in via string-prop wiring (SSM paths, ARNs, URLs). VectorStack between GraphStack and PipelineStack. ScraperStack (Stage 3) likely between PipelineStack and ApiStack or as a co-resource of pipeline.
- AgentLambda construct (infra/constructs/AgentLambda.ts) is the standard pattern for any new agent or scraper Lambda.


SPECIFIC IDEAS
--------------

- Stage 1 demo artifact: running the seed script end-to-end and dumping a Qdrant query result to a markdown file ("show top 5 components for a brief about a luxury bakery"). Proof of life before Stage 2 starts.
- LLM-generated component descriptions — generated once in Stage 1 by Claude with a structured prompt; stored alongside metadata so they're auditable and re-runnable. Not generated at runtime.
- Hybrid retrieval blending in Stage 2 — Composer prompt should make it clear to the LLM where each candidate came from ("from graph: X, Y, Z; from vector retrieval: A, B"). Lets the model weigh signals; supports the "hybrid retrieval" interview narrative.
- Anti-bot reality check for Stage 3 — assume 30-50% of submitted URLs will fail (Cloudflare/Vercel block headless browsers aggressively). Stage 3 needs graceful failure UX in the dashboard ("we couldn't read this URL, paste a different reference?").
- Legal posture for ref sites — UX should frame as "we extract style cues from this site," not "we copy this site." Domain blocklist for major brands (Nike, Apple, etc.) considered for Stage 3.
- Eval rubric direction for Stage 4 — score on (a) brief alignment, (b) component coherence, (c) copy quality. Drives both retrieval ranking improvements and prompt tuning. Specifics decided at Stage 4 planning.


DEFERRED IDEAS
--------------

- CLIP / multi-modal embeddings — visual similarity matching between scraped screenshots and component preview images. Could become a "Stage 5" if the resume needs more weight, but the user chose the learning loop as the strongest closer.
- RAG retrofit to Style / Research / SEO agents — natural follow-up after Stage 4 if the retrieval infrastructure proves valuable. Not in current rollout.
- Past-project embeddings — embedding completed projects (brief → final layout) so Composer can do "past projects similar to this brief used these components." Requires a corpus of completed projects; could be revisited post-Stage-2 once production data accumulates.
- Storybook story scenario embeddings — embed each component story (RealEstate, SaasOnboarding, etc.) as a separate retrieval target. Considered, deferred — start with metadata + LLM descriptions; add stories only if retrieval quality plateaus.
- DDB Streams ingestion Lambda for re-embedding — automatic re-embed when components change. Not needed while you're hand-editing metadata. Add when component churn justifies it (probably post-Stage-4).
- Token-window chunking fallback — only add if real scraped pages prove too messy for semantic chunking. Decided at Stage 3 implementation, not now.
- Local embedding model on Lambda — considered (resume signal), rejected for Stage 1 (cold-start + artifact size pain not worth it). Revisit only if OpenAI cost/latency becomes a problem.
