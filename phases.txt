# Composer RAG v2 — Vector-Only Pivot

Captured 2026-05-04 from a /develop discussion. Stage 2 (hybrid graph + vector
Composer RAG) shipped via PR #29, but a "Luxury Bakery in LA" query returned
3 layout/splits + 1 CTA + 1 hero — no nav, no footer, no text decorator.
Diagnosis: flat top-K cosine over the whole corpus returns whatever's
semantically closest, not "one of each slot the page actually needs." Fixing
this means dropping the graph and reshaping the composer.

This file replaces Decision 5 in `.claude/work/context.md` ("hybrid graph +
vector retriever") with a vector-only path, sequenced into a Foundation PR
plus three follow-up iterations.

----------------------------------------------------------------------------
## Locked decisions (from the discussion)
----------------------------------------------------------------------------

**Architecture**
- Rip Neo4j fully — delete `GraphStack`, the Aura connection, all graph
  retrieval code in the Composer. Update CLAUDE.md to drop the
  "Phase 3 Neo4j" line. Reclaims monthly Aura cost. Narrative becomes
  "evaluated graph + vector, vector won."
- PAIRS_WITH compatibility data stays in `metadata.json` permanently. The
  current `pairsWell` / `pairsPoorly` arrays are the source of truth and
  the Composer reads them as a re-ranking signal post-retrieval. Drop the
  "Until Neo4j is ready..." caveat in `.claude/rules/domains/components.md`.

**Composer flow (plan-then-search-then-pick)**
- Step 1: LLM plans the page skeleton from Brief + Research + Style Agent
  output + optional seller-specified hint ("what kind of page is this?
  landing / store / portfolio").
- Step 2: For each slot in the planned skeleton, vector search Qdrant
  filtered by `category=<slot>` with query
  `"<slot category> for <brief>; mood: <style mood>; needs: <slot purpose>"`.
- Step 3: Re-rank candidates per slot (signals below), then a final LLM
  call assembles the chain from the per-slot top candidates.
- Reference page archetypes (saas-landing, e-commerce, etc.) explicitly
  rejected — too template-y, kills the brief-driven story.

**Re-ranking signals (all four)**
- PAIRS_WITH score — boost candidates whose IDs appear in already-picked
  components' `pairsWell`; demote those in `pairsPoorly`. Greedy
  left-to-right assembly.
- Style overlap — score `component.style[]` / `mood[]` overlap against
  the Style Agent output (similar to today's `agents/composer/handler.ts`
  fallback logic).
- Diversity penalty — penalize chains that over-repeat style/mood across
  consecutive components. Avoids 5 editorial blocks in a row.
- Density / image-weight balance — penalize chains that stack
  high-density-on-high-density or image-heavy-on-image-heavy. Uses
  existing `density` and `imageWeight` metadata fields.

**Embeddings**
- Move to multi-vector per component: ~3 vectors keyed by axis:
  (a) descriptive ("elegant split hero with photography"),
  (b) usage ("hero for premium brands needing strong imagery"),
  (c) audience-fit ("luxury, B2C, visual-first segments").
  Score = max similarity across the 3 vectors.
- LLM-generated descriptions expand from one to three axes:
  "when to use", "audience fit", "design language". Generated by the
  seed script's Claude pass; stored alongside metadata so they're
  auditable and re-runnable. Replaces today's single description.

**Sequencing**
- Foundation PR ships ONE bundled change covering the 4 items in Phase A
  below. The richer-data and richer-flow items in Phases B / C / D ship
  in subsequent PRs — "start the foundations and eventually improve."

----------------------------------------------------------------------------
## Phase A — Foundation PR (vector-only Composer with complete pages)
----------------------------------------------------------------------------

Goal: a query for "Luxury Bakery in LA" returns a complete page (nav +
hero + body sections + CTA + footer), not a flat top-5 dominated by
heroes/splits.

Deliverables (single PR):
1. **Rip Neo4j**
   - Delete `infra/stacks/GraphStack.ts` (or whatever it's called).
   - Drop the `GraphStack` wiring from `MainStage.ts`.
   - Remove graph retrieval from `agents/composer/handler.ts` — the
     existing fallback chain is graph -> DDB; new chain is vector -> DDB.
   - Strip Neo4j SDK / driver dependencies from `agents/package.json`.
   - Update CLAUDE.md (Tech Stack table — drop "Graph (Phase 3): Neo4j
     Aura"; CDK Stacks table — drop GraphStack row).
   - Remove the "Until Neo4j is ready..." line in
     `.claude/rules/domains/components.md` and reframe PAIRS_WITH as
     metadata-native.

2. **Per-slot category-filtered vector search**
   - Replace the Composer's flat top-K with: hard-coded default skeleton
     (Phase A — see #3) -> per-slot Qdrant query filtered by `category=`
     -> top-K candidates per slot.
   - Query template per slot:
     `"<slot category> for <brief>; mood: <style mood>; needs: <slot purpose>"`.
   - Use existing single-vector embeddings for now (multi-vector is
     Phase C). The per-slot filter alone fixes the "no nav in top-5"
     problem.

3. **Hard-coded default skeleton**
   - Default: `nav -> hero -> 2-3 body sections -> CTA -> footer`.
     Body section types tentatively: `services` or `features`,
     `testimonial` or `stats`, optional `faq` or `contact`.
   - Lives in a constant in `agents/composer/` (e.g. `defaultSkeleton.ts`).
   - Phase B replaces this with an LLM planner; the constant stays as the
     fallback when the planner errors.

4. **PAIRS_WITH re-ranking**
   - Read `pairsWell` / `pairsPoorly` from each candidate's metadata
     (already in `ComponentItem` DDB shape — verify on read; otherwise
     pull from manifest at deploy or cache in memory).
   - Greedy left-to-right: once a slot is filled, boost candidates in
     subsequent slots whose IDs appear in any prior pick's `pairsWell`;
     demote those in `pairsPoorly`.
   - This is Phase A's only re-ranking signal. Style overlap, diversity,
     density all land in Phase D.

Demo artifact: re-run `npm run query:vectors --workspace=agents -- "Luxury Bakery in LA"`
(or its successor query script) and confirm the result includes one
component per skeleton slot — nav, hero, body, body, CTA, footer.

Out of scope for Phase A:
- LLM skeleton planner (Phase B)
- Multi-vector per component, 3-axis descriptions (Phase C)
- Style / diversity / density re-ranking (Phase D)

Estimated runway: 4-6 days, single PR.

----------------------------------------------------------------------------
## Phase B — LLM skeleton planner
----------------------------------------------------------------------------

Goal: the Composer stops using a hard-coded skeleton and adapts page
structure to the brief.

Deliverables:
- New LLM call at the head of the Composer: `planSkeleton(brief, research,
  styleOutput, sellerHint?)` -> ordered list of slot categories.
- Inputs: full brief, Research Agent output, Style Agent output (mood,
  density, imageWeight), optional seller-specified page-type hint.
- Output schema: validated with Zod, each slot has `{ category, purpose,
  notes? }`. Categories must be from the canonical 12.
- Falls back to Phase A's hard-coded default skeleton on validation
  failure or LLM error (resilience over correctness).
- Optional new field on the brief / dashboard form: `pageType` (free text
  or enum). Wired through `ProjectBriefSchema` and `PipelineStateSchema`.

Estimated runway: 3-5 days.

----------------------------------------------------------------------------
## Phase C — Multi-vector embeddings + 3-axis descriptions
----------------------------------------------------------------------------

Goal: a query for an obscure niche surfaces the right component because
the embedding corpus covers usage, audience, and design language —
not just descriptive metadata.

Deliverables:
- Re-run the seed script to generate 3 LLM descriptions per component:
  "when to use" / "audience fit" / "design language". Stored alongside
  existing metadata in a new descriptions JSON (auditable, re-runnable).
- Embed each axis as its own vector. Qdrant point per component becomes
  a multi-vector point keyed `descriptive | usage | audience`.
- Composer search: query each axis, score = max similarity across the
  three. Return top-K per slot as before.
- Migration: existing single-vector points get deleted before re-seeding
  (collection wipe + re-seed is fine at current scale — 34 components).

Estimated runway: 3-4 days.

----------------------------------------------------------------------------
## Phase D — Re-ranking signals (style overlap + diversity + density)
----------------------------------------------------------------------------

Goal: the final picked chain is visually balanced, on-brand, and varied.
PAIRS_WITH alone doesn't enforce this.

Deliverables:
- **Style overlap score** — for each candidate per slot, score
  `component.style[]` / `mood[]` overlap with Style Agent output. Add as
  a positive signal alongside PAIRS_WITH.
- **Diversity penalty** — track style/mood frequencies across already-picked
  slots; penalize candidates that over-repeat. Tune threshold empirically
  (e.g. penalize if a tag has been picked 3+ times already).
- **Density / image-weight balance** — penalize stacks of high-density on
  high-density or imageWeight on imageWeight. Use existing metadata
  fields.
- All four signals (PAIRS_WITH + style overlap + diversity + density)
  combine into a single re-ranking score per slot. Weights start equal,
  tuned by Stage 4 eval signal.

Estimated runway: 3-5 days.
